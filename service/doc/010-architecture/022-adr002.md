# ADR-002: Replace Static Enums with Dynamic / Extendable Enum Registries

## Status

Accepted

## Context

The service currently uses Java enums to represent provider types and other extensible concepts.

Adding a new enum value requires a code change, rebuild, and redeployment of the core service. This violates the Open/Closed Principle and makes integration with external partners less flexible.

New values are expected to be introduced by external partners in the future. Multi-tenancy is not a requirement.

---

## Options Considered

### Option 1: Keep Static Java Enums

**Description**

Continue using Java enums for all provider and type definitions.

**Pros**

* ✅ Compile-time type safety
* ✅ Easy refactoring support
* ✅ Simple implementation
* ✅ Automatic OpenAPI enum documentation

**Cons**

* ⬇️ Requires code changes for new values
* ⬇️ Requires rebuild and redeployment
* ⬇️ Not suitable for partner-driven extensions
* ⬇️ Violates Open/Closed Principle for extension points

---

### Option 2: Replace Enums with String-Based Dynamic Registries

**Description**

Replace extensible enums with string-based identifiers validated through Spring-managed registries.

**Pros**

* ✅ Supports extension without modifying existing code
* ✅ Better Open/Closed Principle compliance
* ✅ Suitable for external partner integrations
* ✅ Enables modular and plugin-like extensions

**Cons**

* ⬇️ Reduced compile-time type safety
* ⬇️ Requires runtime validation
* ⬇️ Increased implementation complexity
* ⬇️ Refactoring support is weaker than enums

---

### Option 3: Value Object + Dynamic Registry

**Description**

Replace enums with dedicated value objects (e.g. `ProviderType`) backed by a Spring-managed registry.

**Pros**

* ✅ Extensible without code changes
* ✅ Better type safety than raw strings
* ✅ Clear domain model
* ✅ Supports partner-driven extensions

**Cons**

* ⬇️ More implementation effort
* ⬇️ Requires registry and validation infrastructure
* ⬇️ Slightly higher complexity than raw strings

---

## Decision

Adopt **Option 3: Value Object + Dynamic Registry** for extensible domain concepts.

Extensible types will no longer be represented by Java enums. Instead, they will be modeled as value objects and validated through Spring-managed registries.

This approach provides the required flexibility for external partner extensions while retaining stronger typing and a clearer domain model than a plain string-based solution.

Static Java enums remain acceptable for truly fixed business concepts that are not expected to be extended.

---

## Consequences

### Positive

* New provider types can be added without modifying existing code.
* Supports future partner-driven extensions.
* Better alignment with the Open/Closed Principle.
* Avoids stringly-typed APIs.

### Negative

* Additional registry and validation infrastructure is required.
* Validation shifts from compile-time to application startup/runtime.
* Slight increase in overall complexity.


If you choose the **Value Object + Dynamic Registry** approach from the ADR, the entity would no longer store a Java enum. Instead, it stores a value object (or a string in the database) that is validated against a Spring-managed registry.

## 1. Replace Enum with Value Object

### Before

```java
@Enumerated(EnumType.STRING)
private PriceType priceType;
```

### After

```java
@Convert(converter = PriceTypeConverter.class)
@Column(name = "price_type", length = 100)

```

## 2. Value Object

```java
package io.commercestacksolutions.priceproviderservice.domain.pricetype;

import java.util.Objects;

public record PriceType(String code) {

    public PriceType {
        Objects.requireNonNull(code, "code must not be null");
    }

    @Override
    public String toString() {
        return code;
    }
}
```

Usage:

```java
new PriceType("sales_price");
new PriceType("purchase_price");
new PriceType("custom_partner_price");
```

## 3. JPA Converter

JPA cannot persist a custom value object automatically.

```java
package io.commercestacksolutions.priceproviderservice.dataaccess.pricerow.converter;

import io.commercestacksolutions.priceproviderservice.domain.pricetype.PriceType;
import jakarta.persistence.AttributeConverter;
import jakarta.persistence.Converter;

@Converter(autoApply = true)
public class PriceTypeConverter implements AttributeConverter<PriceType, String> {

    @Override
    public String convertToDatabaseColumn(PriceType attribute) {
        return attribute == null ? null : attribute.code();
    }

    @Override
    public PriceType convertToEntityAttribute(String dbData) {
        return dbData == null ? null : new PriceType(dbData);
    }
}
```

Database values:

```text
sales_price
purchase_price
material_cost
partner_special_price
```

## 6. Registry

Spring automatically discovers all implementations.

```java
package io.commercestacksolutions.priceproviderservice.domain.pricetype;

import org.springframework.stereotype.Component;

import java.util.List;
import java.util.Map;
import java.util.stream.Collectors;

@Component
public class PriceTypeRegistry {

    private final Map<String, PriceTypeDefinition> definitions;

    public PriceTypeRegistry(List<PriceTypeDefinition> definitions) {
        this.definitions = definitions.stream()
                .collect(Collectors.toMap(
                        d -> d.getPriceType().code(),
                        d -> d
                ));
    }

    public boolean exists(String code) {
        return definitions.containsKey(code);
    }

    public PriceTypeDefinition get(String code) {
        return definitions.get(code);
    }

    public void validate(PriceType priceType) {
        if (!exists(priceType.code())) {
            throw new IllegalArgumentException(
                    "Unknown price type: " + priceType.code());
        }
    }
}
```

## 7. Validation Before Save

Example service:

```java
@Service
public class PriceRowService {

    private final PriceTypeRegistry registry;

    public PriceRowService(PriceTypeRegistry registry) {
        this.registry = registry;
    }

    public PriceRowEntity save(PriceRowEntity entity) {

        registry.validate(entity.getPriceType());

        return repository.save(entity);
    }
}
```

## 8. External Partner Extension

A partner module can contribute a new type without modifying existing code:

```java
@Component
public class MarketplacePriceType implements PriceTypeDefinition {

    @Override
    public PriceType getPriceType() {
        return new PriceType("marketplace_price");
    }

    @Override
    public String getDisplayName() {
        return "Marketplace Price";
    }
}
```

No change required in:

* `PriceRowEntity`
* `PriceRowService`
* `PriceTypeRegistry`

The new type is discovered automatically by Spring.
