# ADR: Database Selection and Scalability for Price Provider Service

## Context
The Price Provider service requires a database solution that supports rapid development and testing while providing a path to production-ready scalability. The service needs to handle price-related data efficiently across different environments:

- **Development & Testing**: Fast setup, zero configuration, embedded capabilities
- **Production**: Robust and scalable solution capable of managing high volume of price data, high traffic, and concurrent requests

## Problem Statement
The database selection must balance:
- **Development Speed**: Enable rapid iteration and testing
- **Production Scalability**: Handle escalating transaction per second (TPS) and concurrent access
- **Migration Path**: Smooth transition from development to production databases

## Required Database Capabilities (For Production)
When migrating to a production database, the following capabilities are important for handling high traffic and massive data loads:
* **Scalability and Elasticity**: Ability to scale vertically and horizontally, dynamically adjusting resources based on demand
  * **Replicas Support**: Offloading read operations to replicas for improved query performance
* **Failover and Disaster Recovery**: Redundancy mechanisms ensuring availability during hardware failures or outages

### Possible Future Capabilities (For Production) with Spanner (or Yugabyte or other distributed database)
* **Sharding / Partitioning**: Distributing large datasets across multiple shards/partitions for efficient data management and scalability
* **Read/Write Splitting**: Segregating read and write operations to handle high concurrent read requests

## Constraints
The Price Provider Service is designed as a **starter application template** emphasizing:
- **Simplicity**: Easy to set up and extend
- **Standard Spring Boot**: Leverages standard Spring Boot capabilities and JPA/Hibernate
- **Minimal Dependencies**: H2 for development requires no external setup

For production deployments:
- Database complexity (sharding, read/write splitting, replica management) is handled by the chosen database solution
- Application maintains flexibility within standard Spring Boot features
- Migration from H2 to production database requires only configuration changes (application.yml/application.properties)

## Decision

### Development Stage: H2 Database
For **development and testing purposes**, the service uses **H2 Database**:

**Rationale:**
- **Fast Development Setup**: Embedded in-memory database requires zero external dependencies
- **Fast Testing**: Tests run quickly without external database infrastructure
- **Simplicity**: No installation or configuration required - ideal for a starter template
- **Standard Spring Boot Support**: Full JPA/Hibernate compatibility
- **Easy CI/CD Integration**: No database provisioning needed in build pipelines

### Production Stage: PostgreSQL

For **production environments**, the following database solutions are supported:

#### Primary Candidate: PostgreSQL
- Industry-standard relational database with excellent Spring Boot support
- Strong ACID compliance and proven reliability
- Extensive tooling and community support
- Can be deployed on-premises or as managed service (AWS RDS, Google Cloud SQL, Azure Database)

### Production Stage: Advanced Scalability Option (To Be Evaluated)

#### Advanced Scalability Option: Google Cloud Spanner && yugabyte (www.yugabyte.com) (To Be Evaluated)
Spanner and Yugabyte are candidates for evaluation in scenarios requiring:
- Global distribution and geo-replication
- Horizontal scalability with automatic sharding
- Strong consistency at global scale
- Transactional consistency across regions

**Status**: Requires adpations and proof-of-concept testing to validate this use case. 

### Advanced Scalability Option: Introducing a Caching Layer for Read-Heavy Workloads. (To Be Evaluated)
To further enhance read performance and reduce database load, a caching layer (e.g., Redis, Memcached) could be introduced.
**Status**: Further evaluation required. Measurements and Proof-of-concept implementation required.

## Appendix: Database Options Evaluated / Reviewed

The following database solutions were evaluated for production scalability requirements:

### PostgreSQL (Recommended for Production)
**Technology**: Open-source relational database with proven scalability
**Key Features**:
- Mature ecosystem with extensive Spring Boot support
- ACID compliance and strong data integrity
- Advanced features: JSONB, full-text search, partitioning
- Can be deployed with read replicas and connection pooling
- Managed service options: AWS RDS, Google Cloud SQL, Azure Database for PostgreSQL

**Scalability Approach**:
- Vertical scaling: Increase instance resources
- Horizontal scaling: Read replicas for read-heavy workloads
- Partitioning: Table partitioning for large datasets
- External sharding solutions: Can be combined with middleware like ShardingSphere or Vitess
- Multi Service Setup : Multiple instances of the Price Provider service with separate databases for load distribution

### Google Cloud Spanner (Reviews, To Be Evaluatated)
**Technology**: Fully managed, globally distributed relational database
**Key Features**:
- Transactional consistency at global scale
- Automatic sharding and rebalancing
- Synchronous replication for high availability
- Standard SQL interface
- Seamless horizontal scalability

**Status**: Proof-of-concept testing needed to validate:
- Cost vs. benefit for this use case
- Migration complexity from H2/PostgreSQL
- Integration with Spring Boot and existing data access layer

**Use Cases**: Best suited for:
- Global applications requiring multi-region deployment
- Applications requiring strong consistency across geographies
- Very high scale (millions of TPS)

### Other Options Considered

#### ShardingSphere
- **Technology**: Open-source distributed database middleware
- **Approach**: Proxy layer in front of PostgreSQL/MySQL
- **Features**: Sharding, read/write splitting, distributed transactions
- **Website**: https://shardingsphere.apache.org/

#### Vitess
- **Technology**: Horizontal sharding middleware for MySQL
- **Approach**: Kubernetes-native MySQL sharding
- **Features**: Automatic sharding, distributed transactions, SQL query router
- **Website**: https://vitess.io/

#### MongoDB
- **Technology**: Document-based NoSQL database
- **Features**: Native sharding with auto-balancing, horizontal scalability
- **Limitation**: Requires data model changes (document-oriented vs. relational)

#### CockroachDB
- **Technology**: Distributed SQL database
- **Features**: Horizontal sharding, geo-replication, strong consistency
- **Consideration**: Similar to Spanner but self-hosted or cloud-managed

## Development to Production Migration Path (Proposal)

### Development → Production

1. **Phase 1: Development (Current)**
   - H2 embedded database
   - In-memory or file-based storage
   - Zero configuration required

2. **Phase 2: Testing/Staging**
   - PostgreSQL in containerized environment (Docker)
   - Matches production database type
   - Validates migrations and performance
   - Contiuous Integration / Deployments

3. **Phase 3: Production**
   - PostgreSQL (managed service recommended)
