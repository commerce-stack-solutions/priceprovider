# Bulk Update of Price Rows

This document explains the bulk create-or-update functionality for price rows in the Price Provider Service API.

## Overview

The `POST /admin/api/pricerows/bulk-create-or-update` endpoint provides an intelligent way to create or update multiple price rows in a single operation. This endpoint supports two distinct modes of operation:

1. **Update by ID**: When a price row includes an `id` field, the system updates the existing entity with that ID
2. **Smart Matching**: When no `id` is provided, the system attempts to find a matching price row based on business field values

## Why Smart Matching?

The `id` field is a **technical identifier** generated by the database. In many business scenarios:
- Clients don't know or track these technical IDs
- External systems maintain their own identifiers
- Business logic works with actual price characteristics (product, quantity, currency, etc.)

Smart matching allows clients to update prices based on **business fields** rather than technical IDs. This makes the API more intuitive and reduces the need for clients to maintain ID mappings.

## Update by ID

When you include an `id` field in the price row data, the system will:
- Look up the existing price row by ID
- Update all fields with the provided values
- Return the updated entity

### Example: Update by ID

```json
POST /admin/api/pricerows/bulk-create-or-update
Content-Type: application/json

[
  {
    "id": 123,
    "pricedResourceId": "PRODUCT-001",
    "priceValue": 99.99,
    "minQuantity": 1,
    "unitRef": "piece",
    "currencyRef": "EUR",
    "taxClassRef": "de-vat-full",
    "taxIncluded": false
  }
]
```

**Result**: Price row with ID 123 is updated with the new values.

## Smart Matching (No ID)

When you **omit the `id` field**, the system performs smart matching based on all identifying fields. The system will:

1. Search for an existing price row matching **all** of these fields:
   - `pricedResourceId` - The product/resource identifier
   - `minQuantity` - Minimum quantity for this price
   - `unitRef` - Unit of measurement (e.g., "piece", "kg")
   - `currencyRef` - Currency code (e.g., "EUR", "USD")
   - `taxClassRef` - Tax class identifier
   - `taxIncluded` - Whether tax is included in the price
   - `priceType` - Type of price (e.g., "SALES_PRICE", "PURCHASE_PRICE")
   - `validFrom` - Price validity start date
   - `validTo` - Price validity end date
   - `groupRefs` - Set of group IDs this price applies to (exact match required)

> **Important – Group Matching:** The smart matching algorithm performs an **exact set match** on `groupRefs`.
> This means you must supply **all** groups that the existing price row is assigned to — omitting even one group will cause the system to treat the request as a new price row and create a duplicate instead of updating the existing one.
> For example, if a price row is assigned to `["vip-customers", "newsletter-subscribers"]`, you must provide both IDs in `groupRefs`; providing only `["vip-customers"]` will create a new price row.

> **What Smart Matching Updates:** The smart matching mechanism is designed to update the **price value** (`priceValue`) and/or the **channel assignment** (`channelRefs`) of an existing price row.
> All other identifying fields (unit, currency, tax class, groups, validity dates, etc.) are the matching key and are **not changed** by a smart-match update.
> If you need to change any of the identifying fields, use an ID-based update instead, or create a new price row.

2. If **all fields match** an existing price row:
   - **UPDATE** the existing entity (changes only `priceValue`)
   - Returns the updated entity with its existing ID

3. If **any field differs** (or no match found):
   - **CREATE** a new price row
   - Returns the newly created entity with a new ID

## Examples

### Example 1: Creating a New Price Row

First request creates a new price row:

```json
POST /admin/api/pricerows/bulk-create-or-update
Content-Type: application/json

[
  {
    "pricedResourceId": "PRODUCT-001",
    "priceValue": 99.99,
    "minQuantity": 1,
    "unitRef": "piece",
    "currencyRef": "EUR",
    "taxClassRef": "de-vat-full",
    "taxIncluded": false,
    "priceType": "SALES_PRICE"
  }
]
```

**Result**: New price row created with ID (e.g., 456)

### Example 2: Updating an Existing Price (Smart Matching)

Second request with **same identifying fields** but different price:

```json
POST /admin/api/pricerows/bulk-create-or-update
Content-Type: application/json

[
  {
    "pricedResourceId": "PRODUCT-001",
    "priceValue": 129.99,    // CHANGED PRICE
    "minQuantity": 1,
    "unitRef": "piece",
    "currencyRef": "EUR",
    "taxClassRef": "de-vat-full",
    "taxIncluded": false,
    "priceType": "SALES_PRICE"
  }
]
```

**Result**: Existing price row (ID 456) is **UPDATED** with new price value 129.99

### Example 3: Creating Different Price Tier

Request with **different minQuantity** creates a new tier:

```json
POST /admin/api/pricerows/bulk-create-or-update
Content-Type: application/json

[
  {
    "pricedResourceId": "PRODUCT-001",
    "priceValue": 89.99,
    "minQuantity": 10,        // DIFFERENT QUANTITY
    "unitRef": "piece",
    "currencyRef": "EUR",
    "taxClassRef": "de-vat-full",
    "taxIncluded": false,
    "priceType": "SALES_PRICE"
  }
]
```

**Result**: **NEW** price row created (ID 457) because minQuantity differs

### Example 4: Price with Different Tax Treatment

Request with **different taxIncluded** creates a separate price:

```json

**Result**: **NEW** price row created (ID 458) for tax-inclusive price

### Example 6: Time-Based Pricing

Prices can vary by time period using `validFrom` and `validTo`:

    "validTo": null
  }

```

**Result**: Two separate price rows for different time periods

## Mixed Operations

You can mix ID-based updates and smart matching in a single request:

```json
POST /admin/api/pricerows/bulk-create-or-update
Content-Type: application/json

[
  {
    "id": 123,
    "pricedResourceId": "PRODUCT-001",
    "priceValue": 89.99,
    ...
  },
  {
    "pricedResourceId": "PRODUCT-002",
    "priceValue": 149.99,
    ...
  },
  {
    "pricedResourceId": "PRODUCT-003",
    "priceValue": 199.99,
    ...
  }
]
```

- First entry: Updates by ID 123
- Second entry: Smart matching (update if exists, create if not)
- Third entry: Smart matching (update if exists, create if not)

## Best Practices

### For Price Updates
- **Use smart matching** when you manage prices by business characteristics
- Omit `id` field and provide all identifying fields
- System will find and update the correct price automatically

### For Direct Updates
- **Use ID-based updates** when you need to modify a specific entity
- Include the `id` field
- All other identifying fields can be changed

### For New Prices
- Simply omit the `id` field
- Provide all identifying fields
- System creates a new price if no match is found

### Price Immutability and Legal Compliance

In many business and legal contexts, **price rows should never be modified or deleted after creation**.  Regulatory requirements (e.g. audit trails, bookkeeping rules) may demand a complete, immutable history of every price that was ever in effect.

In such scenarios:

- **Never update or delete existing price rows.**  Treat every price row as an immutable record once it has been applied to an order or invoice.
- **Use `validFrom` / `validTo` to control which price is currently active.**  When a price changes, set the `validTo` of the old price row to the moment the new price takes effect, and create a new price row with the corresponding `validFrom`.
- This pattern ensures a full audit trail: every price that was ever valid can be reconstructed by querying price rows for a given point in time.

**Example – replacing a price without deleting the old one:**

```json
// Step 1: close the existing price row (set validTo via ID-based update)
{ "id": 456, "pricedResourceId": "PRODUCT-001", "priceValue": 99.99, ..., "validTo": "2024-06-30T23:59:59Z" }

// Step 2: create the new price row with validFrom matching the cutover moment
{ "pricedResourceId": "PRODUCT-001", "priceValue": 109.99, ..., "validFrom": "2024-07-01T00:00:00Z", "validTo": null }
```

When immutability is a requirement, **do not use the bulk create-or-update smart matching** to change a price value — use `validFrom`/`validTo` date management instead and rely on smart matching only to keep channel assignments up to date.

## Limitations

- Maximum 100 price rows per request
- All identifying fields must be provided for smart matching to work
- Group references must exist in the system before assignment
- Null values in date fields are treated as "no constraint" for matching

## Error Handling

The endpoint returns partial results:
- Successfully processed entities are returned with their data
- Failed entities are returned with error messages
- HTTP 200 is returned even with partial failures
- Check `$messages` property in response for errors

## Response Example

```json
{
  "items": [
    {
      "id": 456,
      "pricedResourceId": "PRODUCT-001",
      "priceValue": 129.99,
      ...
    },
    {
      "id": 457,
      "pricedResourceId": "PRODUCT-002",
      "$messages": [
        {
          "type": "ERROR",
          "message-key": "common.errors.validation.mandatoryField",
          "message": "Field 'unitRef' is mandatory"
        }
      ]
    }
  ]
}
```

In this example:
- First item was successfully updated/created
- Second item failed validation (missing unitRef)

## Bulk Create-or-Update Matching Strategy per Entity

Different entities use different strategies when processing `bulk-create-or-update` requests:

| Entity | Match Strategy | Key Field(s) |
|--------|---------------|-------------|
| **PriceRow** | **Smart matching** if no `id` provided; **ID match** if `id` is provided | `id` (ID match) or `pricedResourceId` + `minQuantity` + `unitRef` + `currencyRef` + `taxClassRef` + `taxIncluded` + `priceType` + `validFrom` + `validTo` + `groupRefs` (smart match) |
| **Unit** | **Natural key match** by `symbol` | `symbol` (natural key) |
| **Currency** | **Natural key match** by `currencyKey` | `currencyKey` (natural key) |
| **Language** | **Natural key match** by `isoKey` | `isoKey` (natural key) |
| **TaxClass** | **Natural key match** by `taxClassId` | `taxClassId` (natural key) |
| **Group** | **Natural key match** by `id` (group key) | `id` (natural key) |
| **Organization** | **Natural key match** by key | natural key |

### Summary

- **Smart matching** is **only implemented for PriceRow**. PriceRows have a technical auto-generated ID and no unique natural business key, so the system falls back to matching by the combination of all business-relevant fields.
- **All other entities** use their **natural key** (the business identifier like `symbol`, `currencyKey`, `isoKey`) directly – if an entity with that key exists, it is updated; otherwise a new one is created.
